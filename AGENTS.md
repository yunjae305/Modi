# AGENTS.md — AI 에이전트 하네스

> **하네스 철학**: 모델은 엔진이다. 하네스는 핸들, 브레이크, 안전벨트다.  
> 제약이 에이전트를 느리게 만드는 게 아니라, 제약이 없을 때 에이전트가 잘못된 방향으로 빠르게 달린다.

---

## 1. 목적

이 파일은 본 레포지토리에서 작업하는 AI 에이전트의 최상위 실행 계약이다.

- **무엇을**: 작업 규칙, 경계, 검증 루프
- **무엇이 아닌가**: 프로젝트 백과사전, API 문서, 튜토리얼

프로젝트별 세부 규칙은 `docs/` 하위 문서에 분리해서 작성한다.

## 3. 우선순위

지침이 서로 충돌할 경우 다음 우선순위를 따른다.

1. 현재 작업에서 사용자가 명시한 요청
2. 이 `AGENTS.md`
3. `docs/` 하위의 관련 문서
4. 현재 레포지토리의 기존 코드 패턴
5. 일반적인 언어/프레임워크 모범 사례

충돌을 안전하게 해결할 수 없다면 추측하지 말고, 충돌 내용을 보고한 뒤 작업을 중단한다.

## 4. 완전 동작 기본값 ★

> 이 섹션은 이 레포지토리의 핵심 품질 계약이다.

**사용자가 무언가를 만들어달라고 요청하면, 완전히 동작하는 상태로 완성해야 한다.**

### 4.1 완전 동작의 정의

- **실행 가능**: 코드가 실제로 실행되어야 한다. 빌드 에러, 런타임 에러 없음.
- **기능 완성**: TODO, 플레이스홀더, 하드코딩된 더미 데이터 없음.
- **엣지 케이스 처리**: 빈 입력, 에러 상황, 경계값 처리 포함.
- **연결 완성**: 필요한 설정, 환경 변수, 의존성 설치 명령 포함.
- **테스트 가능**: 실행 방법과 확인 방법을 명시.

### 4.2 허용되지 않는 불완전 패턴

다음은 "완성된 코드"가 아니다.

```python
# ❌ 금지
def process_payment():
    # TODO: 실제 결제 로직 구현 필요
    pass

# ❌ 금지
API_KEY = "your-api-key-here"

# ❌ 금지
return {"result": "dummy data"}  # 실제 DB 연동 필요

# ✅ 허용
def process_payment(amount: Decimal, card_token: str) -> PaymentResult:
    """실제 Stripe API를 호출하는 완성된 결제 처리 함수"""
    response = stripe.PaymentIntent.create(
        amount=int(amount * 100),
        currency="usd",
        payment_method=card_token,
    )
    return PaymentResult(id=response.id, status=response.status)
```

### 4.3 완성 전 절대 멈추지 말 것

- 파일 하나 작성하고 "나머지는 비슷하게 구현하면 됩니다" → **금지**
- 핵심 로직만 쓰고 "에러 처리는 추가해주세요" → **금지**
- 프론트엔드만 만들고 "백엔드 API는 별도 구현 필요" (요청 범위 내라면) → **금지**

## 5. 핵심 원칙

### 5.1 아키텍처 환각 금지

아키텍처, 추상화, 디렉토리 구조, 네이밍 규칙, 데이터 흐름, 디자인 패턴을 임의로 만들지 않는다.

반드시 이 레포지토리에 정의된 아키텍처와 기존 패턴을 따른다.

판단이 필요한 경우 레포지토리 문서와 기존 코드를 먼저 읽고, 그래도 불명확하면 추측하지 않고 보고한다.

### 5.2 레포지토리를 단일 진실 공급원으로 사용

레포지토리 문서나 기존 코드가 지침을 제공하는 경우, 외부 지식이나 추측에 의존하지 않는다.

문서, 코드, 테스트, 스키마, 설정 파일을 기준으로 판단한다.

### 5.3 최소 범위 변경

요청된 작업을 완료하는 데 필요한 부분만 수정한다.

관련 없는 리팩터링, 포맷팅 일괄 수정, 의존성 업그레이드, 파일명 변경, 코드 정리를 수행하지 않는다.

### 5.4 기존 동작 보존

명시적 요청 없이 공개 동작, API 계약, 데이터 형식, 설정 키, 권한, 사용자 플로우를 변경하지 않는다.

### 5.5 기계적 검증

코드 변경 후에는 프로젝트에서 정의한 관련 검증 명령을 실행한다.

테스트, 린트, 타입 체크, 빌드를 실제로 실행하지 않았다면 통과했다고 말하지 않는다.

## 6. 3계층 경계 패턴

> GitHub 2,500개 이상 레포지토리 분석 결과 검증된 패턴이다.  
> 에이전트는 작업 전 각 행동이 어느 계층에 해당하는지 분류한다.

| 계층 | 이름 | 정의 | 예시 |
|------|------|------|------|
| 🟢 | **바로 진행** | 범위가 명확하고 되돌릴 수 있는 변경 | 함수 추가, 버그 수정, 테스트 작성 |
| 🟡 | **먼저 질문** | 범위가 불명확하거나 사이드이펙트가 있는 변경 | 새 의존성 추가, 디렉토리 구조 변경, 설정 파일 수정 |
| 🔴 | **절대 금지** | 명시적 승인 없이는 어떤 상황에서도 수행 불가 | DB 마이그레이션 실행, 시크릿 수정, API 계약 파괴적 변경 |

🟡 계층에서 질문할 때는 다음 형식으로 보고한다.

```
[먼저 질문] 다음 작업이 필요합니다:
- 의도: [왜 필요한가]
- 대상: [어떤 파일/시스템]
- 사이드이펙트: [무엇이 영향을 받는가]
- 대안: [다른 방법이 있는가]
진행할까요?
```

## 7. Taste Invariants (품질 불변 조건)

> OpenAI 프로덕션 하네스 시스템에서 도입한 개념이다.  
> 이 규칙들은 CI 실패로 강제되며, "경고"가 아니라 "하드 차단"이다.

이 레포지토리의 Taste Invariants는 `docs/conventions.md`에 정의한다.  
정의가 없는 경우 다음 기본 invariants를 적용한다.

- 함수는 단일 책임을 가진다
- 공개 함수에는 타입 힌트와 docstring이 있어야 한다
- 매직 넘버/스트링은 상수로 추출한다
- 에러는 무시하지 않고 반드시 처리하거나 위로 전파한다
- 로그 메시지에는 컨텍스트(ID, 상태값)가 포함된다

## 8. 컨텍스트 라우터

코드를 수정하기 전에 작업과 관련된 문서를 먼저 읽는다.

### 8.1 빌드 및 테스트

읽을 문서: `docs/build.md`

확인할 내용:
- 설치 명령
- **파일 단위 테스트 명령** (전체 스위트보다 빠른 피드백)
- 전체 테스트 명령
- 린트, 타입 체크, 포매터 명령
- 빌드 명령
- CI 요구사항

`docs/build.md`가 없다면 다음 설정 파일을 확인한다.
- `package.json` / `pyproject.toml` / `Makefile`
- CI 설정 파일 (`.github/workflows/`, `.gitlab-ci.yml`)
- lockfile, 프레임워크 설정 파일

근거 없이 명령을 임의로 만들지 않는다.

### 8.2 아키텍처

읽을 문서: `docs/architecture.md`

확인할 내용:
- 레이어 경계 및 의존성 방향
- 모듈 책임
- 디렉토리 구조
- 서비스/컴포넌트 위치
- 네이밍 규칙

역방향 의존성을 만들거나 기존 경계를 우회하지 않는다.

### 8.3 코딩 컨벤션

읽을 문서: `docs/conventions.md`

확인할 내용:
- 포맷팅 스타일
- 네이밍 규칙
- 에러 처리 방식
- 로깅 방식
- import/export 패턴
- 테스트 스타일
- 파일 구성 방식

### 8.4 API 계약

읽을 위치: `docs/api/`

명시적인 승인 없이 API 계약을 변경하지 않는다.

### 8.5 데이터베이스 및 영속성

관련 작업일 때 읽을 항목:
- `docs/database.md`
- migration 파일, schema 파일, model 정의, seed 파일

명시적인 승인과 마이그레이션 경로 없이 파괴적인 변경을 만들지 않는다.

### 8.6 보안

관련 작업일 때 읽을 문서: `docs/security.md`

항상 지킬 규칙:
- 시크릿, 토큰, API 키, 개인 키, 쿠키, 인증 정보를 노출하지 않는다
- 인증 또는 인가를 약화시키지 않는다
- 검증, 권한 체크, rate limit, 암호화를 우회하지 않는다
- 명시적 지시 없이 프로덕션 시크릿이나 `.env` 값을 수정하지 않는다
- 예시 값은 placeholder를 사용한다

## 9. 에이전트 실행 루프 (PEV — Plan · Execute · Verify)

> PEV는 단일 패스가 아니다. 검증 실패 시 Execute → Verify 사이클을 반복한다.

### 9.1 Plan (계획)

수정 전에 다음을 읽고 내부 계획을 수립한다.

읽어야 할 것:
- 사용자 요청
- 관련 문서 (섹션 8 컨텍스트 라우터 기준)
- 수정 대상 파일과 주변 구현
- 관련 테스트 및 스키마

계획 항목:
- 의도한 변경
- 3계층 경계 분류 (섹션 6)
- 수정 가능성이 있는 파일
- 실행할 검증 명령 목록
- 위험 요소 또는 불확실한 부분

계획은 사용자에게 출력하지 않는다. 내부 판단 기준으로만 사용한다.  
단, 🟡 계층 행동이 필요하거나 불확실한 부분이 있으면 작업 전에 보고한다.

### 9.2 Execute (실행)

가장 작고 정확한 변경을 수행한다.

- 기존 패턴을 따른다
- 관련 없는 수정을 피한다
- 완전 동작 기본값(섹션 4)을 지킨다
- 동작이 변경되는 경우 테스트를 추가하거나 수정한다

### 9.3 Verify (검증 루프)

검증은 단순 실행이 아니라 통과할 때까지 반복하는 루프다.

**검증 순서 (파일 단위 → 전체)**

```
1. 파일 단위 테스트   (수정한 모듈만, 가장 빠른 피드백)
2. 통합 테스트        (관련 모듈 범위)
3. 타입 체크
4. 린트
5. 포매터 체크
6. 전체 빌드
```

앞 단계가 실패하면 다음 단계로 진행하지 않는다.

**실패 분류**

| 분류 | 판단 기준 | 처리 방식 |
|------|-----------|-----------|
| 내 변경에 의한 실패 | 수정한 파일/모듈과 직접 관련 | 수정 후 루프 재시작 |
| 기존에 존재하던 실패 | 수정 전부터 있던 문제 | 기록하고 계속 진행 |
| 판단 불가 | 원인 파악 불가 | 중단하고 사용자에게 보고 |

**재시도 한계**

동일한 검증 항목에서 3회 초과 실패 시 중단한다.

```
시도 1 → 실패 → 원인 분석 → 수정
시도 2 → 실패 → 원인 분석 → 수정
시도 3 → 실패 → 중단 및 보고
```

**롤백 조건**

다음 중 하나라도 해당하면 변경을 롤백하고 사용자에게 보고한다.
- 3회 시도 후에도 실패가 해결되지 않는 경우
- 실패 원인이 요청 범위를 벗어난 구조적 문제인 경우
- 수정 시도가 기존 동작을 예상치 않게 변경하는 경우
- 원인을 파악할 수 없는 실패가 발생한 경우

**검증 흐름**

```
검증 명령 실행
    │
    ├─ 통과 → 다음 검증 단계
    │
    └─ 실패 → 분류
                ├─ 내 변경 → 시도 < 3 → 수정 → 재실행
                │         └─ 시도 ≥ 3 → 롤백 → 보고
                ├─ 기존 실패 → 기록 후 계속
                └─ 판단 불가 → 롤백 → 보고
```

## 10. 금지 사항

AI 에이전트는 다음 행동을 해서는 안 된다.

- 문서화되지 않은 아키텍처를 임의로 만들기
- TODO, 플레이스홀더, 더미 데이터로 "완성된 척" 하기
- 관련 없는 리팩터링 수행
- 승인 없이 API 계약 변경
- 명확한 필요 없이 의존성 추가
- 검증 통과를 위해 테스트 삭제 또는 우회
- 원인 해결 없이 에러 숨기기
- 실패한 검증 명령 무시
- 시크릿 또는 인증 정보 노출
- 기존 프로젝트 스타일을 개인 선호 스타일로 교체
- 요구사항이 불명확한 상태에서 크고 되돌리기 어려운 변경 수행
- 재시도 한계를 초과한 상태에서 계속 시도하기
- "나머지는 유사하게 구현하면 됩니다" 류의 미완성 인계

## 11. 문서 누락 시 처리 규칙

참조 문서가 없거나 불완전한 경우 다음을 따른다.

1. 추측하지 않는다
2. 주변 코드, 테스트, 스키마, 설정 파일을 확인한다
3. 가장 가까운 기존 패턴을 따른다 (패턴을 따르는 것은 추측이 아니라 관찰이다)
4. 변경은 작고 되돌릴 수 있게 유지한다
5. 문서가 없거나 부족했다는 점을 보고한다

## 12. 완료 기준

작업은 다음 조건을 모두 만족해야 완료된 것으로 간주한다.

1. 요청된 변경이 구현되었다
2. **코드가 실제로 실행 가능한 완전한 상태다** (섹션 4)
3. 관련 문서와 기존 패턴을 따랐다
4. 필요한 경우 테스트를 추가하거나 수정했다
5. 검증 루프를 완료했다 (모든 관련 검증이 통과했거나 실패 사유가 명확히 보고되었다)
6. 재시도가 있었다면 시도 횟수와 원인을 기록했다
7. 최종 응답이 실제 수행 내용을 구체적이고 정직하게 설명한다

## 13. 최종 응답 형식

작업 완료 보고에는 다음 형식을 사용한다.  
단순 작업(파일 1개, 재시도 없음)은 불필요한 항목을 생략할 수 있다.

```text
요약:
- ...

변경된 파일:
- path/to/file.py  [추가 / 수정 / 삭제]

검증:
- `명령어`: 통과 / 실패 / 생략
- `명령어`: 통과 / 실패 / 생략 (N회 시도)

완전 동작 확인:
- 실행 방법: `...`
- 확인 방법: `...`

기존 실패 (발견된 경우):
- ...

메모:
- ...
```
